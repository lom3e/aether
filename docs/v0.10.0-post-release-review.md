# Architectural Review — v0.10.0 Planning & Context Runtime

**Data:** 2026-07-18  
**Stato:** Post-Release Architectural Review  
**Milestone di riferimento:** v0.10.0 (Planning & Context Runtime)  
**Autore:** Senior Software Architect

---

## 1. Stato Finale dell'Architettura (v0.10.0)

Con il completamento della milestone v0.10.0, l'architettura di Aether si è evoluta per supportare l'esecuzione iterativa e dinamica (ReAct). I componenti principali presentano il seguente stato:

*   **Agent**: Gestisce l'identità, il ciclo di vita (`lifecycle`) e controlla direttamente il **ReAct Loop** (ciclo iterativo multi-turno). Ha la responsabilità di validare i limiti di sicurezza dell'esecuzione (`max_turns`, `max_tool_calls`, `max_total_tokens`) e di mappare i tool disponibili nel registro locale nel JSON Schema atteso dai provider.
*   **AgentContext**: Nuovo modello dati mutabile definito in `src/aether/core/execution.py`. Eredita da `ExecutionContext` evitando la duplicazione dello stato e memorizza esclusivamente lo stato temporaneo e locale dell'esecuzione corrente:
    *   La cronologia dei messaggi (`messages: list[Message]`).
    *   Il consumo cumulato di token (`token_usage: dict[str, int]`).
    *   Lo stato di esecuzione e il turno corrente (`current_turn: int`).
    *   `AgentContext` è del tutto privo di logiche attive (non conosce tool, skill, esecutori o registri).
*   **ExecutionEngine**: Funge da puro confine di esecuzione. Oltre a supportare l'esecuzione di piani statici (`build_plan` + `run` per retrocompatibilità v0.9.0), espone il metodo `execute_tool_calls(tool_calls, context)` per risolvere ed eseguire dinamicamente i tool a runtime senza compilazione preventiva del piano.
*   **Provider Layer**: Basato sull'astrazione `AIProvider` e le sue implementazioni (`MockProvider`, `OllamaProvider`). Si occupa esclusivamente di tradurre il protocollo di rete e i formati dati nativi dell'LLM nei contratti unificati di Aether.
*   **Contratti `ToolCall` e `ToolResult`**: Definiti in `src/aether/core/execution.py` come interfacce core. Rappresentano in modo provider-agnostic l'intenzione dell'LLM di invocare un tool (`ToolCall`) ed il relativo output (`ToolResult`).
*   **Skill System**: Integrato come comportamento deterministico eseguito all'inizio del task tramite piano statico (garantendo retrocompatibilità al 100% con la v0.9.0).

## 2. Verifica Separazione delle Responsabilità

La delimitazione dei confini architetturali è stata rigorosamente rispettata:

1.  **L'Agente non esegue direttamente i Tool**: L'Agente rileva le intenzioni di chiamata ma delega l'esecuzione fisica all'`ExecutionEngine`, mantenendo separata la logica strategica da quella operativa.
2.  **L'ExecutionEngine non gestisce lo stato conversazionale**: L'engine riceve il contesto come parametro di sola lettura e non prende decisioni sul flusso del loop, agendo come mero esecutore sincrono.
3.  **Il Provider non conosce i registri locali**: Il provider riceve una lista passiva di schemi di tool ed una sequenza di messaggi strutturati. Non ha visibilità su come i tool siano registrati o implementati in locale.
4.  **Il Core è Provider-Agnostic**: Le strutture dati in `src/aether/core/` non importano moduli dai provider e rimangono conformi agli standard di astrazione del framework.

## 3. Analisi di Problemi Tecnici e Debito Tecnico

L'ispezione del codice post-release evidenzia le seguenti aree di miglioramento o potenziale debito tecnico:

### A. Cablaggio del JSON Schema dei Tool nell'Agente
Attualmente, `Agent.execute()` contiene un blocco di codice procedurale che compila manualmente il dizionario JSON Schema per ciascun tool abilitato:
```python
tools_schema.append({
    "type": "function",
    "function": {
        "name": tool.name,
        "description": tool.description,
        "parameters": { ... }
    }
})
```
*   **Problema**: Questo approccio appesantisce il codice dell'Agente e lo accoppia a una specifica rappresentazione dei parametri dei tool (funzioni a singolo parametro stringa `input_data`).
*   **Mitigazione**: Delegare la serializzazione dello schema all'interfaccia `Tool` stessa (es. implementando una property `Tool.schema` o un metodo `Tool.to_json_schema()`), lasciando all'Agente solo il compito di collezionarli.

### B. Gestione asimmetrica dei tipi in `ToolCall.arguments`
*   **Problema**: A seconda del provider o dello stato del loop, `arguments` può essere ricevuto come dizionario (es. Ollama native) o stringa JSON. L'`ExecutionEngine` esegue una risoluzione euristica in `execute_tool_calls` per estrarre la stringa `input_data`. 
*   **Mitigazione**: Uniformare il contratto: `ToolCall.arguments` deve essere sempre un dizionario Python strutturato. Qualsiasi serializzazione/deserializzazione a stringa JSON deve essere gestita esclusivamente al confine del Provider.

### C. Loop ReAct sincrono e monolitico in `Agent.execute`
*   **Problema**: La logica del ciclo ReAct (invocazione provider -> parsing -> esecuzione tool -> accumulo -> controlli di sicurezza) è implementata in un unico ciclo `while True` sincrono all'interno del metodo principale dell'agente. Se si desiderasse introdurre lo streaming, l'esecuzione asincrona o un tipo di loop differente (es. Plan-and-Solve), il metodo diventerebbe ingestibile.
*   **Mitigazione**: Estrarre la logica del loop in un orchestratore dedicato (es. `ReActOrchestrator` o `AgentRunner`) o rendere configurabile il "runner" dell'agente.

## 4. Valutazione della Preparazione per Funzionalità Future

### A. Memory Runtime
*   **Stato**: Ottimo. `AgentContext` traccia lo storico dei messaggi turn-by-turn. Un futuro runtime di memoria (es. a breve termine o semantica) potrà innestarsi prima o dopo ogni invocazione del provider per arricchire il contesto o persistere le informazioni.

### B. Multi-Agent Orchestration
*   **Stato**: Ottimo. Poiché gli agenti Aether implementano l'interfaccia comune `Agent.execute(Task, ExecutionContext) -> ExecutionResult`, qualsiasi agente può essere registrato come `Tool` all'interno del `ToolRegistry` di un agente coordinatore (Orchestratore), realizzando gerarchie di esecuzione isolate e ricorsive senza modifiche al runtime.

### C. Streaming delle Risposte
*   **Stato**: Richiede refactoring. La firma sincrona attuale di `AIProvider.generate()` e il loop sincrono dell'Agente bloccano il thread corrente fino a generazione ultimata. Il supporto allo streaming richiederà l'introduzione di firme asincrone (`async`/`await`) e generatori asincroni (`AsyncIterator`), impattando l'intera catena di esecuzione.

### D. Supporto OpenAI / Ollama Tools
*   **Stato**: Pronto. La presenza di `tool_calls` e `tool_call_id` in `Message` ricalca accuratamente la specifica OpenAI. Le correzioni introdotte nell'`OllamaProvider` (conversione a dizionario dei parametri) dimostrano che i provider possono mappare e negoziare le peculiarità dei vari LLM in modo trasparente.

## 5. Proposte e Roadmap per la Milestone v0.11.0

### Cosa NON Modificare
1.  **Immutabilità di `ExecutionContext`**: Mantenere la separazione tra il contesto statico immutabile (`ExecutionContext`) e lo stato dinamico mutabile (`AgentContext`).
2.  **Responsabilità dell'ExecutionEngine**: Mantenere l'engine come esecutore puro privo di logiche decisionali o di conservazione di stato globale.
3.  **Contratti Core**: Non spostare `ToolCall`, `ToolResult` o `Message` fuori dal layer `aether.core`.

### Cosa Migliorare
1.  **Refactoring della Serializzazione dei Tool**: Rimuovere la compilazione del JSON schema dei tool da `Agent` e implementare `Tool.to_json_schema()` o simile in `aether.tools.base.Tool`.
2.  **Predisposizione Asincrona**: Valutare l'introduzione preliminare di firme asincrone per gli esecutori e i provider per facilitare lo streaming.

### Architettura consigliata per la Milestone v0.11.0: Memory Runtime
L'obiettivo della v0.11.0 sarà l'implementazione del **Memory Runtime**. Si consiglia la seguente struttura:

```text
                     AgentContext (Conversational Memory)
                               ▲
                               │
            Agent ──[Interroga/Persiste]──> MemoryManager
                                                │
                       ┌────────────────────────┴────────────────────────┐
                       ▼                                                 ▼
        ConversationMemory (Short-Term)                   SemanticMemory (Long-Term / Vector)
```

1.  **MemoryManager**: Componente centrale per il coordinamento della memoria, iniettabile nell'Agente.
2.  **ConversationMemory**: Gestione automatica del caricamento/salvataggio dello storico, strategie di riassunto (summarization) ed eliminazione (truncation) quando si supera il limite di token memorizzato nell'`AgentContext`.
3.  **SemanticMemory**: Interfaccia per l'archiviazione e la ricerca semantica di fatti rilevanti (ad es. tramite embeddings o semplici database chiave-valore).
