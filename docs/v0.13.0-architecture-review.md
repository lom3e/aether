# Architecture Review — v0.13.0 Agent Communication & Coordination Runtime

**Data:** 2026-07-18  
**Stato:** Proposta Architetturale  
**Milestone di riferimento:** v0.13.0 (Agent Communication & Coordination Runtime)  
**Autore:** Senior Software Architect

---

## 1. Obiettivi

La milestone **v0.13.0** evolve il supporto multi-agent introdotto nella v0.12.0 da semplice delegazione gerarchica sincrona a un **runtime di coordinamento** con:

1.  **Agent Communication Layer**: Un message bus locale per il routing dei messaggi tra agenti.
2.  **Task Coordination**: Lifecycle formale dei task delegati con stati tracciabili.
3.  **Orchestration Runtime**: Un orchestratore strutturato che supera il pattern `AgentTool` diretto.
4.  **Event System**: Contratti per eventi di sistema (agent started, task completed, ecc.).
5.  **Context Management**: Regole esplicite per la condivisione controllata di contesto tra agenti.

---

## 2. Stato Attuale (Post v0.12.0)

```text
Agent
  ├── AgentContext (effimero, per-run)
  ├── ExecutionEngine (tool dispatch)
  ├── MemoryManager (optional)
  └── ToolRegistry
        └── AgentTool → Agent figlio (isolamento task_id)

AgentRegistry → resolve / search_by_role / search_by_capability
AgentMessage  → contratto dati (sender, receiver, parent_task_id)
DelegationContext → chain tracking, depth/circularity safety
```

### Limiti attuali
*   La comunicazione tra agenti avviene solo implicitamente tramite `AgentTool.execute()` → non c'è un canale di messaggistica esplicito.
*   Non esiste un lifecycle formale per i task delegati: il task è "running" o "done", senza stati intermedi.
*   L'orchestratore è un semplice agente con `AgentTool` nei propri tools — non ha visibilità strutturata sui risultati intermedi.
*   Non esiste un sistema di eventi per monitorare cosa accade nel runtime.

---

## 3. Principi Architetturali

*   **Local-first**: Il message bus è un oggetto Python in-process, non un servizio esterno. Nessuna dipendenza da broker, code, o infrastrutture distribuite.
*   **Provider-agnostic**: Il coordination layer non ha dipendenze dai provider LLM.
*   **Modulare**: Ogni componente (bus, task tracker, event emitter) è opzionale e iniettabile. Un agente singolo senza bus, senza tracker e senza eventi continua a funzionare identicamente a v0.12.0.
*   **Zero over-engineering**: Niente pattern pub/sub distribuiti, niente serializzazione remota, niente async runtime. Tutto sincrono e in-process nella v0.13.0.

## 4. Nuovi Componenti

### A. AgentMessageBus (`src/aether/coordination/message_bus.py`)

Un bus di messaggi locale in-process per il routing esplicito di `AgentMessage` tra agenti registrati.

**Responsabilità:**
*   Registrare handler per agente (nome) in modo che un agente possa ricevere messaggi.
*   Inviare un `AgentMessage` a un destinatario specifico, invocandone l'handler.
*   Mantenere un log ordinato dei messaggi scambiati per tracciabilità.

**Design:**
*   Modello sincrono: `send()` invoca l'handler immediatamente e ritorna il risultato.
*   Nessuna coda asincrona, nessun thread pool.
*   L'handler di default è una semplice callable `(AgentMessage) -> AgentMessage | None`.

```python
bus = AgentMessageBus()
bus.register("Worker", handler_fn)
response = bus.send(AgentMessage(sender="Orchestrator", receiver="Worker", content="..."))
```

### B. TaskState & TaskTracker (`src/aether/coordination/task_tracker.py`)

Un tracker leggero per il lifecycle dei task delegati.

**TaskState** (enum):
*   `CREATED` → task creato, non ancora assegnato.
*   `ASSIGNED` → task assegnato a un agente.
*   `RUNNING` → task in esecuzione.
*   `COMPLETED` → task completato con successo.
*   `FAILED` → task terminato con errore.

**TaskTracker:**
*   `create(task_id, parent_task_id)` → registra un task come `CREATED`.
*   `transition(task_id, new_state)` → effettua la transizione di stato con validazione (le transizioni non valide sollevano errore).
*   `get_state(task_id)` → restituisce lo stato corrente.
*   `get_children(parent_task_id)` → restituisce i sotto-task di un task padre.
*   `get_history(task_id)` → restituisce la cronologia delle transizioni.

### C. Event System (`src/aether/coordination/events.py`)

Contratti per eventi osservabili nel runtime.

**AgentEvent** (base dataclass):
*   `event_type`: tipo di evento (enum).
*   `agent_name`: agente sorgente.
*   `task_id`: task di riferimento.
*   `timestamp`: data e ora dell'evento.
*   `metadata`: dati aggiuntivi.

**EventType** (enum):
*   `AGENT_STARTED`
*   `TASK_DELEGATED`
*   `TASK_COMPLETED`
*   `AGENT_FAILED`

**EventEmitter:**
*   `on(event_type, callback)` → registra un listener.
*   `emit(event)` → invoca tutti i listener registrati per quel tipo di evento.
*   Sincrono, in-process, nessuna coda.

### D. Coordinator (`src/aether/coordination/coordinator.py`)

Un orchestratore strutturato che evolve il pattern `AgentTool`.

**Responsabilità:**
*   Ricevere un task primario e un `AgentRegistry`.
*   Delegare sotto-task ad agenti specifici tramite il `AgentMessageBus`.
*   Tracciare il lifecycle dei sotto-task tramite il `TaskTracker`.
*   Emettere eventi tramite l'`EventEmitter`.
*   Raccogliere e aggregare i risultati.

**Design:**
Il `Coordinator` non sostituisce l'`Agent`. È un componente complementare iniettabile che può essere utilizzato da un agente orchestratore per coordinare più agenti, ma l'agente stesso continua a funzionare tramite il proprio ReAct loop e provider.

## 5. Context Management

### Regola Base: Isolamento per Default
Ogni delegazione continua a creare un contesto completamente isolato (come in v0.12.0):
*   Nuovo `Task` con `task_id` univoco.
*   Nuovo `AgentContext` effimero.
*   `ConversationMemory` indicizzata per `task_id` → nessuna collisione.

### Condivisione Controllata (Opzionale)
Il `Coordinator` può passare selettivamente informazioni al sotto-agente tramite il campo `metadata` del `Task`. Il contesto del padre **non viene mai copiato integralmente** nel figlio.

Esempio:
```python
sub_task = Task(
    agent_name="Worker",
    instruction="...",
    metadata={"shared_context": {"project": "Aether", "priority": "high"}},
)
```

L'agente figlio accede ai dati condivisi tramite `task.metadata["shared_context"]`, ma non ha accesso ai messaggi, ai token usage o alla conversazione del padre.

## 6. Compatibilità Architetturale

| Componente | Stato | Impatto v0.13.0 |
|---|---|---|
| Agent (v0.10) | Invariato | Nessuna modifica al core agent |
| AgentContext (v0.10) | Invariato | Rimane effimero e per-run |
| MemoryManager (v0.11) | Invariato | Nessuna modifica |
| AgentTool (v0.12) | Invariato | Continua a funzionare per deleghe semplici |
| AgentRegistry (v0.12) | Invariato | Utilizzato dal Coordinator per risolvere agenti |
| DelegationContext (v0.12) | Invariato | Riutilizzato dal Coordinator per la safety |
| AgentMessage (v0.12) | Esteso | Riutilizzato come payload del MessageBus |

Nessun componente esistente viene modificato. I nuovi componenti sono opzionali e additivi.

## 7. Rischi e Decisioni

### Rischio 1: Complessità dell'Orchestratore
Il Coordinator potrebbe accumulare responsabilità e diventare un "god object".
*   *Decisione*: Il Coordinator si limita a coordinare bus + tracker + emitter. La logica di decisione su "quali agenti coinvolgere" rimane nell'LLM dell'agente orchestratore tramite il ReAct loop.

### Rischio 2: Performance con Molti Agenti
La comunicazione sincrona potrebbe diventare un collo di bottiglia con catene di delega profonde.
*   *Decisione*: Nella v0.13.0 il runtime è sincrono. L'introduzione di esecuzione concorrente o asincrona è rimandata a una milestone futura (v0.14.0+), quando il pattern sarà validato.

### Rischio 3: Accoppiamento tra Coordinator e Agent
Il Coordinator non deve forzare modifiche all'`Agent`.
*   *Decisione*: Il Coordinator opera esclusivamente tramite interfacce pubbliche esistenti (`Agent.execute(task)`, `AgentRegistry.resolve()`, `AgentMessage`). Non accede ad attributi interni dell'Agent.
