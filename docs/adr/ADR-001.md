# ADR-001: Unified Execution Contract

## Date

2026-07-16

## Status

Accepted

---

## Context

In Aether, capabilities were historically divided. Skills had a formal runtime contract (`SkillResult` / `SkillExecutionStatus`), with lifecycle validation, error categorization, and hooks. Tools followed a completely separate path, returning unstructured data and throwing raw exceptions like `KeyError` directly to the execution pipeline.

## Problem

Splitting execution results between Skills and Tools created a duplicate pipeline. The `Agent` had to branch its code depending on whether it was executing a skill or a tool, leading to complex and error-prone orchestrations. It also blocked building a generic Execution Engine that could treat skills and tools as uniform execution units.

## Decision

Unify the execution contract of both Skills and Tools under a single data structure: `UnitExecutionResult`.
*   Both Skill and Tool executions return a `UnitExecutionResult`.
*   Introduce the `UnitType` enum (containing `SKILL` and `TOOL`) to replace string-based type checks.
*   Alias the legacy `SkillResult` to `UnitExecutionResult` to preserve backward compatibility.
*   Enforce that all executors catch runtime errors and return a `UnitExecutionResult` with `status = UnitExecutionStatus.FAILED` instead of letting raw exceptions bubble up.

## Consequences

### Positive
*   A single, standardized execution pipeline.
*   No branching logic in the engine or agents.
*   Strong typing prevents silent bugs from string checks.

### Negative
*   Legacy tests checking `result.unit_type == "skill"` needed to be updated to enum comparisons.

## Future Implications

By sharing a unified execution result contract, it becomes easy to support new capability types in the future (e.g. MCP tools, nested agents, plugins) simply by registering them under a new `UnitType` enum value and returning the same `UnitExecutionResult` schema.
