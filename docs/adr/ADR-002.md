# ADR-002: ExecutionEngine as Runtime Orchestrator

## Date

2026-07-16

## Status

Accepted

---

## Context

Prior to v0.8.0, the `Agent` held all runtime execution orchestration responsibilities. Its `execute()` loop directly iterated over skills, verified compatibility, handled tool execution branching, implemented the fail-fast policy, and managed the final prompt synthesis.

## Problem

Entrusting the `Agent` with runtime execution orchestration created a "leaky" abstraction:
1.  The Agent became a heavy component with multiple responsibilities: identity, reasoning/LLM communication, and low-level loop management.
2.  Testing the execution flow required instantiating the full Agent hierarchy.
3.  It was impossible to run agents with custom execution dynamics without subclassing the Agent itself.

## Decision

Move all execution orchestration responsibilities from `Agent` to a dedicated `ExecutionEngine`.
*   The `ExecutionEngine` owns the execution loop, error trapping, unit dispatch, and result aggregation.
*   The `Agent` delegates execution to `ExecutionEngine.run()`.
*   Make `ExecutionEngine` fully injectable in the `Agent` constructor, allowing custom engine overrides.

## Consequences

### Positive
*   Clearer separation of concerns: Agent handles Reasoning and Identity; Engine handles Execution.
*   Improved testability: Execution logic can be tested directly on the `ExecutionEngine` using mock/in-memory contexts.
*   Decoupled architecture: Third-party applications can customize agent execution by injecting a tailored engine.

### Negative
*   Introduces an extra abstraction layer between the Agent and the actual executors (Skills/Tools).

## Future Implications

With the execution logic encapsulated in `ExecutionEngine`, we can easily introduce advanced execution behaviors in subsequent milestones—such as concurrent execution, event hooks, or sandboxed executions—by modifying the engine without touching the Agent code.
