# ADR-003: ExecutionPlan Foundation

## Date

2026-07-16

## Status

Accepted

---

## Context

In previous versions of Aether, there was no separate step to determine what an agent was about to do before doing it. The execution loop resolved capabilities on the fly as it progressed through task execution.

## Problem

Without a representation of the execution plan, the system suffered from several limitations:
1.  **Dry-Runs / Validation**: It was impossible to review or validate the list of actions an agent intended to perform before running them.
2.  **No Introspection**: Users or parent orchestrators could not inspect the execution flow.
3.  **LLM Planning Integration**: Integrating an LLM-based planner was difficult because there was no data contract to represent a series of actions generated by a model.

## Decision

Introduce the `ExecutionPlan` and `ExecutionPlanState` as a foundational data contract.
*   The `ExecutionPlan` contains an ordered list of `ExecutionUnit`s (e.g. `SkillUnit`, `ToolUnit`) representing the actions to be taken.
*   The `ExecutionPlan` tracks execution state (PENDING, RUNNING, COMPLETED, FAILED) and stores the resulting list of `UnitExecutionResult`s.
*   The engine separates building a plan (`build_plan()`) from executing a plan (`run()`).

## Consequences

### Positive
*   Decoupled planning from execution.
*   Enables full runtime visibility, logging, and inspection of plans.
*   Establishes the exact data structures needed to implement an autonomous Planning Engine in later milestones.

### Negative
*   Introduces additional model classes (`ExecutionPlan`, `SkillUnit`, `ToolUnit`) that must be populated and passed around.

## Future Implications

In future milestones, we can introduce a Planning Engine where an LLM analyzes the instruction, constructs an `ExecutionPlan` with dynamic dependencies, and passes it to the `ExecutionEngine` for execution. This ADR ensures that the execution layer is ready for planning without requiring structural changes.
