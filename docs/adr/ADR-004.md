# ADR-004: Message-Based Provider Interface

## Date

2026-07-17

## Status

Accepted

---

## Context

In Aether v0.2.0, the provider interface was established with a minimal `generate(prompt: str) -> str` signature. This was adequate for a test scaffold, but incompatible with the message format required by modern LLM APIs (OpenAI, Anthropic, Ollama, Gemini), which all use an array of `{role, content}` message objects.

## Problem

Passing a concatenated string to LLM APIs has several critical issues:
1. **No System / User separation**: LLM behavior is heavily influenced by the `system` role. Conflating instructions with user input degrades model quality.
2. **No Structured Context**: Memory recall and tool outputs need to be injected with semantic clarity; raw string concatenation is ambiguous.
3. **No Rich Output**: Returning only a `str` loses critical metadata: token usage (for cost tracking), `finish_reason` (to detect truncation or tool calls), and model identity.

## Decision

Replace the `generate(prompt: str) -> str` interface with `generate(messages: list[Message]) -> ProviderResponse`.

Introduce three data contracts:
*   **`Message(role, content)`**: A single conversation turn.
*   **`ProviderConfig`**: Configuration (model, api_key, base_url, temperature, timeout, retries).
*   **`ProviderResponse`**: Standardized output (content, model, usage dict, finish_reason).

The `Agent` is updated to construct a `list[Message]` before calling the provider, mapping its role to a `system` message, and injecting memory context and tool results as additional `system` messages.

## Consequences

### Positive
*   Compatible with OpenAI, Anthropic, Ollama, Gemini APIs out of the box.
*   Rich provider metadata (usage, model, finish_reason) available in `ExecutionResult`.
*   Clear separation between system instructions and user input.

### Negative
*   Breaking change for anyone who implemented `AIProvider` with the old `str -> str` signature.

## Future Implications

The `finish_reason` field in `ProviderResponse` enables future support for tool calling detection (when the model returns `finish_reason = "tool_calls"`). The `ProviderConfig` can be extended with a `stream: bool` field to support streaming responses without changing the base interface contract.
