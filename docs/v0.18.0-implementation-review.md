# v0.18.0 — Agent Delegation Layer Implementation Review

## Obiettivo Milestone
Introdurre un meccanismo di **Agent Delegation** pulito, scalabile e rigorosamente ancorato all'architettura cognitiva di Aether, consentendo a un agente genitore (tramite il Planner) di delegare parte del suo `Goal` a un agente specializzato (tramite Tool), mantenendo la totale immutabilità dell'`ExecutionEngine`. 

## Architettura Implementata
L'architettura ha seguito scrupolosamente il paradigma *Agent as Tool*, introducendo lo **Structured Intelligence Layer (v0.17.0)** nella dinamica di delega.
1. Il Planner genitore, usando output strutturati, compone una `DelegationRequest` al nuovo `CognitiveAgentTool`.
2. `CognitiveAgentTool` intercetta la richiesta, converte i parametri JSON in un `Goal`, popola le properties (success criteria, constraints) ed esegue `child_agent.achieve(goal)`.
3. Il ciclo cognitivo del child_agent si risolve isolatamente, restituendo un `ExecutionResult`.
4. L'`ExecutionResult` viene riconfezionato in un `DelegationResult` immutabile e restituito strutturalmente, affinché l'ExecutionEngine genitore continui a processare il risultato (oggetto Any, non stringa!) e lo riproponga fedelmente all'Observation per il genitore.

## File Modificati / Introdotti
- `src/aether/core/delegation.py` [NEW]: Modulo estratto per ospitare `DelegationContext` e la sua protezione da loop infiniti di delega (es. `Agent A -> Agent B -> Agent A`).
- `src/aether/core/communication.py` [MOD]: Rimosso `DelegationContext`, aggiunto il re-export di backward compatibility.
- `src/aether/planning/delegation.py` [NEW]: Introdotte le classi `DelegationRequest` e `DelegationResult` (cognitive layer schema).
- `src/aether/tools/cognitive_agent_tool.py` [NEW]: Adapter cognitivo (`CognitiveAgentTool`) che instrada una delega sotto forma di `Goal`.
- `tests/test_agent_delegation.py` [NEW]: Aggiunti 6 test dedicati alla catena di esecuzione.

## Decisioni Importanti
1. **DelegationContext via Dependency Injection**: Si è evitato di far decidere profondità di delega e catena al `Tool`. Il Tool è un puro esecutore passivo. È compito del runtime genitore fornire un `DelegationContext`, e il Tool propaga l'aggiornamento.
2. **DelegationResult come Struct**: A differenza degli execution standard testuali, è emerso che `DelegationResult.output` non doveva essere castato a stringa (JSON stringify). Il Tool restituisce un `Any`, lasciando al framework la responsabilità di trattarlo (fondamentale per permettere la ricezione intatta di dict/JSON provenienti da sub-agenti).
3. **Nessun Modifica all'ExecutionEngine**: Zero reference a `CognitiveAgentTool` nel motore. Trattato come una qualsiasi funzione Python.

## Test Eseguiti (169 Test Passati)
Sono stati introdotti nuovi test dedicati che coprono:
- Propagazione del contesto (constraint, success criteria).
- Isolamento memorie `ConversationMemory`.
- Loop protection (raise DelegationError su circoli ricorsivi).
- Legacy compatibility per `AgentTool` (old execution path).
- Garanzia di conservazione del tipo (Structured Delegation Result).

## Compatibilità Mantenuta
- `Agent.execute()` e `Agent.achieve()`: Non modificate (0 reference a delegazioni o sub-agenti).
- `ExecutionEngine`: Nessuna modifica strutturale o referenza ciclica.
- Progetti che usano `AgentTool` o importano da `communication` continuano a girare via re-export temporanei.

## Rischi Futuri
1. Un Planner troppo generoso nella delega potrebbe consumare velocemente il limite di Max Depth. 
2. Da monitorare l'estensione del budget di token dell'agente genitore, dato che un `DelegationResult` molto corposo (in termini di JSON restituito dal child agent) si riverserà interamente nell'`Observation` dell'agente genitore. In future release potrebbe essere utile una logica di sumarizzazione o embedding della memoria.
