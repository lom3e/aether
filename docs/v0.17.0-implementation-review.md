# Aether v0.17.0 — Implementation Review

## Obiettivo della Milestone
La milestone v0.17.0 introduce lo **Structured Intelligence Layer**, dotando il framework di capacità per supportare output strutturati dai provider e un meccanismo di validazione dei piani cognitivi.

## Componenti Implementati

### 1. Provider Capabilities
- Creata l'astrazione immutabile `ProviderCapabilities` per definire esplicitamente il supporto a strumenti come `tools`, `vision`, `structured_output` e `thinking`.
- Esposta la `property capabilities` read-only su `AIProvider`.
- Aggiornato `OllamaProvider` per riflettere le sue capacità effettive (tools, structured output, e thinking).

### 2. Supporto Output Strutturato
- Aggiornata la firma del metodo `generate` nei provider con il parametro opzionale `output_schema: Any | None = None`, permettendo l'invio di schema JSON o classi Pydantic.
- Implementata la logica per convertire elegantemente l' `output_schema` nel parametro `format` per le API Ollama.
- Aggiornato `BasicPlanner.generate_plan` per accettare e inoltrare l'`output_schema` opzionale al provider.

### 3. Plan Validation Layer
- Introdotta la classe astratta `PlanValidator` con il suo output immutabile `ValidationResult` nel package `aether.planning.validation`.
- Integrato un ciclo di fallback sicuro in `Agent.achieve()`. Se il piano generato fallisce la validazione pre-esecuzione (`is_valid=False`), l'agente evita la compilazione verso il runtime esecutivo (`ExecutionEngine`).

## Test
Tutti i componenti sopra descritti sono stati documentati e dotati di suite di test dedicate nel file `tests/test_structured_intelligence.py`.
Inoltre, è stata eseguita una massiccia correzione dei mock providers della suite esistente per renderli compatibili con l'aggiornamento della firma astratta. L'ExecutionEngine e il metodo `Agent.execute(Task)` mantengono **piena backward compatibility** per l'uso legacy (zero fallimenti registrati).

## Compatibilità Architetturale
Il layer cognitivo mantiene un netto disaccoppiamento dal layer di execution. Nessuna God-class o over-engineering sono stati introdotti; le API estese per lo schema strutturato restano del tutto opzionali, senza appesantire l'astrazione core.

## Hardening Post-Audit: Validation Failure Decision Flow (fix)

### Problema identificato
L'audit finale della v0.17.0 aveva rilevato un rischio architetturale **ALTO**: quando `PlanValidator.validate()` restituiva `ValidationResult(is_valid=False)`, l'`Agent.achieve()` costruiva autonomamente una `Decision(REPLAN)` senza coinvolgere il Planner. Questo causava:

1. Il Planner non riceveva alcuna informazione sul motivo del fallimento
2. Lo stesso identico piano veniva rigenerato ad ogni iterazione
3. Il ciclo `while True` non aveva possibilità di terminazione naturale

### Soluzione adottata: `BasePlanner.evaluate_validation_result()`

**Decisione architetturale**: aggiungere su `BasePlanner` un metodo concreto non astratto:

```python
def evaluate_validation_result(
    self,
    result: ValidationResult,
    goal: Goal,
    plan: CognitivePlan,
) -> Decision:
    ...
```

**Motivazioni della scelta:**

- `ValidationResult` e `Observation` restano **semanticamente distinti**. `Observation` = esito di un'azione reale a runtime. `ValidationResult` = evento pre-esecuzione cognitivo. Due contratti diversi, due metodi diversi.
- Il metodo è **concreto, non astratto**: le implementazioni esistenti di `BasePlanner` non si rompono (backward compatibility totale). Il default REPLAN con reasoning sugli errori è sempre corretto.
- **Estendibile per future milestone**: planner avanzati (v0.18+ swarms, provider LLM multipli) possono override con strategie di recovery intelligenti (constraint relaxation, fallback tool selection, ecc.).
- Il `Agent.achieve()` torna ad essere un **puro orchestratore**: passa informazioni tra componenti senza interpretare eventi cognitivi.

### Nuovo flusso validation failure

```
CognitivePlan
 |
 v
PlanValidator.validate()
 |
 v
ValidationResult(is_valid=False)
 |
 v
planner.evaluate_validation_result(result, goal, plan)
 |
 v
Decision(action=REPLAN, reasoning=..., metadata={"validation_errors": [...]})
 |
 +-- se REPLAN  --> continue (rigenerazione piano)
 +-- se FINISH/CONTINUE --> ExecutionResult(success=False)  [safeguard]
```

### File modificati
- `src/aether/planning/planner.py`: aggiunto `evaluate_validation_result` su `BasePlanner`
- `src/aether/agents/agent.py`: `Agent.achieve()` delega la decisione al Planner
- `src/aether/planning/__init__.py`: esportati `PlanValidator` e `ValidationResult` dalla public API
- `tests/test_structured_intelligence.py`: aggiunti test dedicati per tutti i casi
