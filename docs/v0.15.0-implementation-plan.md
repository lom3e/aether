# Implementation Plan - v0.15.0 Observability & Runtime Diagnostics

Questo documento descrive il piano implementativo ordinato per la milestone v0.15.0.
Tutto il nuovo codice sarà contenuto nel nuovo package `src/aether/observability/`.

## Step 1: Modelli Dati e Strutture Core

1. Creare `src/aether/observability/trace.py`.
2. Implementare la classe `TraceEvent` come `dataclass`.
   - Campi: `event_id`, `trace_id`, `task_id`, `parent_task_id`, `timestamp` (`datetime`), `event_type` (str/Enum estensibile), `component` (str), `metadata` (dict).
   - L'uso del tempo persitente deve sfruttare unicamente `datetime.now(timezone.utc)`.
3. Implementare la classe `ExecutionMetrics` come `dataclass`.
   - Campi numerici inizializzati a zero: `total_duration_ms`, `provider_time_ms`, `tool_time_ms`, `total_tokens`, `retry_count`, `timeout_count`, `error_count`, `successful_tasks`, `failed_tasks`.
4. Implementare la classe `RuntimeTrace`.
   - Contenitore **immutabile** di eventi. Fornisce costruttori per l'inizializzazione statica, o funge da output read-only.
   - Non deve mutare `ExecutionMetrics` internamente durante la sua creazione, piuttosto esporre un metodo `compute_metrics() -> ExecutionMetrics` o dipendere da un `MetricsBuilder` futuro.
   - Metodi: `to_dict()` (restituisce dict), `to_json()` (restituisce stringa JSON formattata).

## Step 2: Collector In-Memory

1. Creare `src/aether/observability/collector.py`.
2. Implementare la classe `TraceCollector`.
   - Struttura dati interna: dizionario thread-safe (usando `threading.Lock`) che mappa `trace_id` -> lista di `TraceEvent`.
   - **Attenzione**: Documentare e garantire che i lock vengano usati unicamente per proteggere le operazioni di mutazione del dizionario e non per la serializzazione.
   - Metodi:
     - `add_event(event: TraceEvent)`: Inserisce in modo sicuro un evento.
     - `get_trace(trace_id: str) -> RuntimeTrace`: Crea e restituisce un'istanza immutabile di `RuntimeTrace` data la lista di eventi raccolti.
     - `remove(trace_id: str)`: Rilascia la memoria per la trace indicata (usato alla fine dell'esecuzione del task).
     - `clear()`: Pulisce la memoria globale del collector.

## Step 3: Facciata Diagnostics e Integrazione Eventi

1. Creare `src/aether/observability/diagnostics.py`.
2. Implementare la classe `RuntimeDiagnostics`.
   - Modello ad **istanza** (non singleton globale). 
   - Detiene un riferimento al `TraceCollector`.
   - Implementa logica per esportare tracce in un file JSON formattato sfruttando `RuntimeTrace.to_json()`.
3. Creare metodi bridge (es. `attach_to_emitter(emitter)`) in `RuntimeDiagnostics`.
   - La facciata può registrarsi sull'`EventEmitter` esistente per intercettare `AGENT_STARTED`, `TASK_COMPLETED`, etc. e tradurli automaticamente in `TraceEvent`.
   - Calcolo delle durate temporali (se necessario per `tool_time_ms` o task interni) dovrà sfruttare unicamente `time.perf_counter()`.

## Step 4: Arricchimento Opzionale per Provider e Tool

*(Nota: Questo step sarà valutato durante l'implementazione per assicurarsi di non violare il principio di non modifica ai componenti esistenti, favorendo pattern di decorazione o estensione innocua).*
1. Valutare l'emissione di eventi granulari (es. `PROVIDER_CALLED`, `TOOL_CALLED`) nel flusso dell'`ExecutionEngine` / `Agent`.
2. Tali eventi dovrebbero contenere i dati necessari per popolare `ExecutionMetrics` tramite calcoli ex-post in `RuntimeTrace.compute_metrics()`.

## Step 5: Test Unitari

1. Aggiungere la directory `tests/test_observability.py`.
2. Test da scrivere:
   - Creazione e popolamento immutabile di un `RuntimeTrace` e calcolo metriche tramite `compute_metrics()`.
   - Esportazione corretta chiamando `to_dict()` e `to_json()`.
   - Thread-safety del `TraceCollector` inserendo eventi da multipli thread simultanei assicurandosi che il lock funzioni.
   - Comportamento corretto di `remove()` su `TraceCollector`.
   - Verifica di utilizzo esclusivo di `datetime.now(timezone.utc)` e `time.perf_counter()`.
