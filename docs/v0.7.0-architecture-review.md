# Architecture Review — v0.7.0 Unified Execution Runtime

**Data:** 2026-07-16
**Branch:** `feature/skill-execution-runtime`
**Test suite:** 32/32 passed

---

## 1. ExecutionEngine

### È realmente il punto centrale di esecuzione?

**Parzialmente.** L'`ExecutionEngine` raccoglie la delegazione verso `SkillExecutor` e `ToolExecutor` in un unico punto, il che è corretto. Tuttavia, non è ancora il *punto centrale* dell'esecuzione dell'agente: l'`Agent.execute()` contiene ancora tutto il loop sulle skill, la costruzione del prompt, il raccolta del contesto di memoria e la chiamata al provider. L'engine è usato come *helper*, non come *orchestratore primario*.

Il ciclo in `Agent.execute()` è ancora questo:

```python
for skill in active_skills:
    skill_result = self.execution_engine.execute_skill(skill, execution_context)
    if not skill_result.success:
        # Build failure, return
        ...
```

L'`Agent` conosce ancora il concetto di "loop di skill", mentre idealmente questo dovrebbe avvenire all'interno dell'engine stesso.

**Impatto architetturale:** Medio. Non bloccante per v0.7, ma in v0.8 o v0.9 (quando arriveranno Planning Engine o Multi-Agent), l'agent dovrà cedere ulteriormente responsabilità all'engine. Il loop corrente diventa un problema quando il set di skill non è più statico.

### L'astrazione è sufficientemente generica per future estensioni?

Sì, per ora. L'engine espone `execute_skill` e `execute_tool` come metodi di primo livello. È semplice aggiungere in futuro `execute_plan` o `execute_action` senza rompere l'API esistente.

### Ci sono duplicazioni con SkillExecutor o ToolExecutor?

No duplicazioni dirette. L'engine è correttamente un layer di facciata. L'unica cosa da monitorare è il fatto che il parametro `override_registry` in `execute_tool` lega l'engine alla logica di risoluzione del registry che l'`Agent` gestisce autonomamente:

```python
result = self.execution_engine.execute_tool(
    tool_name, tool_input, tool_context,
    override_registry=registry,  # registry preso dal context, non dall'engine
)
```

Questo è un accoppiamento implicito: l'`Agent` bypassa la configurazione interna dell'engine per passare un registry alternativo. Nel tempo, quando il tool registry dell'engine e quello del context divergono, questo diventa una fonte di bug.

## 2. Skill e Tool unificati

### La separazione tra Skill e Tool è corretta?

Sì, la separazione concettuale è corretta e ben mantenuta:

- **Skill**: Unità declarativa, validata rispetto allo stato del lifecycle dell'agente. Non produce I/O esterno.
- **Tool**: Unità imperativa con effetti collaterali su sistemi esterni. Non ha lifecycle validation.

Questi due concetti hanno responsabilità distinte e la scelta di non forzare un'interfaccia comune tra `Skill` e `Tool` è corretta per questa fase.

### La nuova struttura permette future capability più complesse?

Sì, ma con un avvertimento: `Tool.execute()` restituisce ancora `str`. Questo vincola l'output strutturato. Se in futuro un tool deve restituire JSON strutturato, metadati di paginazione o stream di dati, `str` non è sufficiente. Il `ToolExecutor` salva la situazione nel breve termine avvolgendo l'output in `UnitExecutionResult.output: Any`, ma la firma di `Tool.execute()` dovrebbe essere pensata prima di introdurre tool complessi.

## 3. UnitExecutionResult

### Il contratto è sufficientemente generico?

Per v0.7, sì. I campi coprono le esigenze immediate. La tabella seguente analizza la completezza rispetto ai requisiti futuri previsti:

| Campo | Presente | Note |
|---|---|---|
| `unit_id`, `unit_name`, `unit_type` | ✅ | Corretto |
| `status` (enum) | ✅ | Corretto |
| `unit_version` | ✅ | Corretto |
| `output` | ✅ | Corretto |
| `error`, `error_type` | ✅ | Corretto |
| `execution_time_ms` | ✅ | Telemetria base |
| `retry_count` / `retry_attempt` | ❌ | Non presente |
| `timeout_triggered` | ❌ | Non presente |
| `trace_id` / `span_id` | ❌ | Non presente per distributed tracing |
| `logged_at` / timestamp | ❌ | Non presente |

I campi mancanti **non sono bloccanti per v0.7** e la loro assenza è giustificata dall'obiettivo dichiarato di non over-engineerare. Sono però da pianificare prima di introdurre observability (v0.9+ presumibilmente).

### Sostituisce correttamente SkillResult?

Tecnicamente sì, tramite aliasing:

```python
SkillResult = UnitExecutionResult
SkillExecutionStatus = UnitExecutionStatus
```

Tuttavia, `UnitExecutionResult` usa `slots=True` ed è un `dataclass`. `SkillResult = UnitExecutionResult` non è una vera sottoclasse, è un riferimento allo stesso tipo. Questo significa che `isinstance(result, SkillResult)` funziona, ma `type(result) is SkillResult` darà `False` per chiunque usi `SkillResult` come factory. Potenzialmente fonte di confusione nel futuro, ma non un problema ora.

Le property di compatibilità (`skill_id`, `skill_name`, `skill_version`) sull'oggetto base sono un'eccezione accettabile: sono esplicite e documentate nel codice con il commento `# Backward compatibility properties for SkillResult`. Da rimuovere quando si elimineranno i riferimenti a `SkillResult` nei test.

## 4. Agent Architecture

### L'Agent è solo orchestratore, lifecycle manager e provider coordinator?

**Quasi, ma non del tutto.** Analisi per responsabilità:

| Responsabilità | Corretto? |
|---|---|
| Lifecycle management (`initialize`, `execute`, `fail`, `complete`) | ✅ |
| Provider coordination (`self.provider.generate(prompt)`) | ✅ |
| Context building (`_build_context`) | ✅ — leggermente verboso, ma accettabile |
| Prompt building (`_build_prompt`) | ⚠️ Borderline — logica di composizione del prompt è applicativa |
| Skill resolution (`resolve_skills`, `_resolve_canonical_skill`) | ⚠️ Duplica logica già in `SkillExecutor.resolve_skill` |
| Memory collection (`_collect_memory_context`) | ⚠️ L'Agent conosce il formato della stringa di memoria |
| Tool execution routing (`_execute_requested_tool`) | ⚠️ L'Agent conosce il formato dei metadati del Task |

**Problema principale:** `_execute_requested_tool` legge `task.metadata["tool_name"]` e `task.metadata["tool_input"]` direttamente. Questo significa che l'`Agent` conosce una convenzione di naming dei metadati del `Task`. Quando arriveranno altri trigger di tool (es. da output LLM, da planning engine), questa logica non scala.

Anche `_build_prompt` è un metodo applicativo che conosce il dominio (come formattare memory, tool output e instruction in una stringa). Questa responsabilità dovrebbe risiedere in un futuro `PromptBuilder`.

**Verdict:** l'Agent ha ancora 2-3 responsabilità applicative che sopravviveranno fino a v0.8/v0.9. Non sono bloccanti, ma devono essere messe nel radar.

## 5. Future Compatibility

| Scenario | Compatibilità | Note |
|---|---|---|
| **Planning Engine** | 🟡 Parziale | L'engine ha i metodi giusti, ma il loop di skill nell'Agent deve essere estratto nell'engine |
| **Memory System** | 🟡 Parziale | `ExecutionContext` porta già `Memory`, ma l'Agent la gestisce direttamente anziché delegar la al contesto |
| **Multi-Agent Runtime** | 🟡 Parziale | Ogni `Agent` ha il proprio `ExecutionEngine` istanziato nell'`__init__`. In multi-agent, l'engine dovrebbe essere condivisibile o configurabile dall'esterno |
| **Async execution** | 🟡 Parziale | Tutto è sincrono ora. `Tool.execute()` e `Skill.execute()` (futuro) sono bloccanti. La firma attuale è compatibile con un wrapper `run_in_executor` ma non è stata progettata per async nativo |
| **Event System** | 🔴 Non pronto | Non c'è alcun hook di pubblicazione di eventi nel risultato dell'esecuzione. Un tool o skill che completa con successo non notifica nessun bus |
| **Plugin System** | 🟡 Parziale | `SkillExecutionHooks` è un embrione di plugin system. Ma non esiste equivalente per tool o per l'engine stesso |

## 6. Problemi Nascosti

### 6.1 Coupling implicito tra Agent e ExecutionEngine tramite `override_registry`

Il parametro `override_registry` in `ExecutionEngine.execute_tool` è un pattern che indica che l'engine non è l'unica sorgente di verità per la configurazione. L'`Agent` ha il suo `tool_registry`, l'`ExecutionContext` può averne un altro, e l'engine ha il proprio. Tre fonti potenzialmente divergenti per la stessa risorsa.

### 6.2 `Agent.__init__` costruisce un `ExecutionEngine` con configurazione parziale

```python
self.execution_engine = ExecutionEngine(
    skill_executor=SkillExecutor(registry=self.skill_registry),
    tool_registry=self.tool_registry,
)
```

Il `ToolExecutor` interno all'engine è quello default. Se si vuole iniettare un `ToolExecutor` customizzato (es. con retry, timeout), non c'è modo farlo dall'esterno senza modificare l'`Agent`. L'`ExecutionEngine` non è iniettabile dall'esterno dell'agent.

### 6.3 `unit_type` come stringa libera

```python
unit_type: str  # "skill" | "tool"
```

Usare una stringa libera anziché un `Enum` è una scelta pragmatica ora, ma introduce un potenziale bug silenzioso: `unit_type="Skill"` non causerebbe errori ma romperebbe qualsiasi confronto.

### 6.4 `_execute_requested_tool` risolleva `KeyError` senza wrapping

```python
except KeyError:
    raise
```

Un `KeyError` da una chiave mancante nel registry risale fino al chiamante non tipizzato. Dovrebbe essere convertito in un `UnitExecutionResult` con `status=FAILED` per mantenere il contratto uniforme.

## 8. Verdict Finale

### Stato architettura: **READY con osservazioni**

L'architettura di v0.7.0 è internamente coerente, non contiene over-engineering e mantiene la complessità sotto controllo. Le fondamenta del runtime unificato sono solide.

### Problemi bloccanti per il rilascio di v0.7.0

Nessuno. La suite è verde, i contratti pubblici sono retrocompatibili, e le astrazioni introdotte sono reversibili.

### Problemi da risolvere prima o durante v0.8.0

1. **`unit_type` come `Enum`** — basso costo, elimina una categoria di bug silenzioso.
2. **`ExecutionEngine` iniettabile nell'`Agent`** — necessario per testabilità avanzata e multi-agent.
3. **`_execute_requested_tool` deve restituire `UnitExecutionResult` anziché risollevare `KeyError`** — necessario per consistenza del contratto.
4. **Il loop di skill deve migrare nell'`ExecutionEngine`** — prerequisito per Planning Engine.

### Si può procedere a v0.8.0?

**Sì.** L'architettura v0.7.0 è sufficientemente stabile per procedere. I problemi identificati non sono debito tecnico accidentale: sono trade-off consapevoli, documentati, e reversibili nella prossima milestone.
