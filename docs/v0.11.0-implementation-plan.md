# Piano di Implementazione — v0.11.0 Memory Runtime

Questo documento definisce il piano ufficiale di implementazione per la milestone **v0.11.0 — Memory Runtime**, finalizzata all'introduzione della ritenzione conversazionale (short-term) e semantica (long-term) in Aether.

---

## 1. Obiettivo della Milestone

L'obiettivo della milestone v0.11.0 è estendere il runtime di Aether abilitando la ritenzione dello storico della conversazione tra più sessioni e turni, la protezione della finestra di contesto del modello tramite politiche di truncation, e l'arricchimento dei prompt con fatti estratti dalla memoria a lungo termine.

---

## 2. Step Implementativi Ordinati

### Step 1: Contratti e Astrazioni base della Memoria
*   **Obiettivo**: Definire le interfacce per la memoria in `src/aether/memory/base.py`.
*   **Dettagli**:
    *   Definire la classe astratta `BaseMemoryStore` con metodi di lettura/scrittura comuni.
    *   Definire il modello dati `MemoryDocument` per incapsulare testo, metadata e timestamps dei fatti a lungo termine.

### Step 2: Implementazione di `ConversationMemory` (Short-Term)
*   **Obiettivo**: Implementare lo store per lo storico dei messaggi in `src/aether/memory/conversation.py`.
*   **Dettagli**:
    *   Fornire una classe `ConversationMemory` che gestisce il salvataggio dei messaggi legati a una sessione (task o utente).
    *   Implementare la logica di **sliding window** e di **truncation**: se il conteggio cumulato dei token in `AgentContext` supera una soglia configurata, rimuovere i messaggi più vecchi garantendo la preservazione del prompt di sistema e dell'ultimo prompt utente.

### Step 3: Implementazione di `SemanticMemory` (Long-Term)
*   **Obiettivo**: Creare la memoria semantica a lungo termine in `src/aether/memory/semantic.py`.
*   **Dettagli**:
    *   Fornire una classe `SemanticMemory` per l'archiviazione e la query di `MemoryDocument`.
    *   Implementare un meccanismo locale di persistenza e ricerca testuale basato su SQLite o file JSON.
    *   Implementare una metrica di similitudine non vettoriale (es. TF-IDF semplice o distanza di Jaccard sui token) per il retrieval senza dipendenze pesanti.

### Step 4: Sviluppo del `MemoryManager`
*   **Obiettivo**: Creare l'orchestratore centrale in `src/aether/memory/manager.py`.
*   **Dettagli**:
    *   La classe `MemoryManager` incapsula `ConversationMemory` e `SemanticMemory`.
    *   Espone metodi come `load_context(context: AgentContext) -> None` per caricare lo storico e i fatti semantici pertinenti prima dell'esibizione dei messaggi al provider.
    *   Espone `persist_context(context: AgentContext) -> None` per salvare lo storico e dedurre nuovi fatti al completamento.

### Step 5: Integrazione con l'Agent Loop
*   **Obiettivo**: Cablare il `MemoryManager` nel ciclo ReAct di `Agent`.
*   **Dettagli**:
    *   Modificare `Agent.__init__` per accettare un `memory_manager: MemoryManager | None = None`.
    *   In `Agent.execute()`, invocare `memory_manager.load_context()` per popolare l'iniziale `AgentContext.messages` con lo storico conversazionale e i fatti rilevanti prima dell'avvio del loop.
    *   Ad ogni turno del loop, verificare se è necessaria la truncation dei messaggi se si rischia di sforare il limite di token.
    *   Alla conclusione dell'esecuzione, invocare `memory_manager.persist_context()` per archiviare lo storico.
