# Piano di Implementazione — v0.11.0 Memory Runtime

Questo documento definisce il piano ufficiale di implementazione per la milestone **v0.11.0 — Memory Runtime**, finalizzata all'introduzione della ritenzione conversazionale (short-term) e semantica (long-term) in Aether.

---

## 1. Obiettivo della Milestone

L'obiettivo della milestone v0.11.0 è estendere il runtime di Aether abilitando la ritenzione dello storico della conversazione tra più sessioni e turni, la protezione della finestra di contesto del modello tramite politiche di truncation, e l'arricchimento dei prompt con fatti estratti dalla memoria a lungo termine.

### Limiti dello Scope (Out of Scope per v0.11.0)
Non sono inclusi in questa milestone (rimandati a milestone future):
*   Embeddings e calcolo vettoriale.
*   Integrazione con database vettoriali.
*   Strategie di summarization automatica tramite chiamate LLM secondarie.
*   Estrazione automatica dei fatti a fine esecuzione tramite LLM.

### Vincoli Architetturali
1.  **AgentContext Stateless ed Effimero**: L'oggetto `AgentContext` deve rimanere privo di logiche di persistenza, I/O o database. Agisce solo come contenitore temporaneo di stato a runtime.
2.  **Migrazione di `Agent.memory`**: L'attuale attributo `Agent.memory` (legacy) verrà gradualmente deprecato e sostituito da un `MemoryManager` esplicito, garantendo la compatibilità a ritroso (alias/wrapper compatibili per evitare breaking changes).

## 2. Step Implementativi Ordinati

### Step 1: Contratti e Astrazioni base della Memoria (`src/aether/memory/base.py`)
*   **Dettagli**:
    *   Definire la classe astratta `BaseMemoryStore` con metodi di lettura/scrittura comuni.
    *   Definire il modello dati `MemoryDocument` per incapsulare testo, metadata e timestamps dei fatti a lungo termine.

### Step 2: Implementazione di `ConversationMemory` (Short-Term) (`src/aether/memory/conversation.py`)
*   **Dettagli**:
    *   Fornire una classe `ConversationMemory` per registrare lo storico completo dei messaggi (user, assistant, tool).
    *   Implementare la logica di **sliding window** e di **truncation**: se il conteggio cumulato dei token in `AgentContext` supera una soglia definita, rimuovere i messaggi intermedi più vecchi garantendo la preservazione del prompt di sistema iniziale e dell'ultimo prompt dell'utente.

### Step 3: Implementazione di `SemanticMemory` (Long-Term) (`src/aether/memory/semantic.py`)
*   **Dettagli**:
    *   Fornire una classe `SemanticMemory` per l'archiviazione locale di `MemoryDocument` basata su SQLite o file JSON.
    *   Implementare una metrica di similitudine testuale semplice non vettoriale (es. keyword matching o TF-IDF/Jaccard minimale sui token) per il retrieval locale-first.

### Step 4: Sviluppo del `MemoryManager` (`src/aether/memory/manager.py`)
*   **Dettagli**:
    *   La classe `MemoryManager` incapsula `ConversationMemory` e `SemanticMemory`.
    *   Fornisce le API per caricare lo storico e recuperare i fatti semantici pertinenti prima del ReAct loop (`load_context`), e salvare lo storico a fine esecuzione (`persist_context`).

### Step 5: Integrazione con l'Agent Loop e Retrocompatibilità (`src/aether/agents/agent.py`)
*   **Dettagli**:
    *   Modificare `Agent.__init__` per accettare un parametro `memory_manager: MemoryManager | None = None`.
    *   Sostituire internamente l'uso dell'attributo legacy `Agent.memory` con il `MemoryManager`, implementando proprietà/wrapper di compatibilità per mantenere il funzionamento del codice preesistente che referenziava `Agent.memory`.
    *   Nel loop `Agent._run_loop`, invocare `memory_manager.load_context()` prima del loop ed eseguire la truncation durante il loop se i token superano la soglia. A fine esecuzione, invocare `memory_manager.persist_context()`.
