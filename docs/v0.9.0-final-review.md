# Aether v0.9.0 — Provider Runtime Integration Final Review

**Date:** 2026-07-17
**Status:** READY FOR TAG

---

## 1. Stato Architettura Post-Implementazione

Il layer provider di Aether è stato completamente modernizzato. Il pattern **string-in / string-out** del provider foundation (v0.2.0) è stato sostituito da un contratto strutturato e ricco.

### Struttura finale del provider layer

```
src/aether/providers/
├── __init__.py       — public API exports
├── base.py           — AIProvider abstract class (updated)
├── types.py          — Message, ProviderConfig, ProviderResponse [NEW]
├── errors.py         — ProviderError hierarchy [NEW]
├── manager.py        — ProviderManager registry [NEW]
├── mock.py           — MockProvider (updated to new contract)
└── ollama.py         — OllamaProvider (real LLM integration) [NEW]
```

### Flusso di esecuzione aggiornato

```
Runtime
  ↓
Agent  (lifecycle + context + message building)
  ↓
ExecutionEngine  (plan + dispatch)
  ↓
Execution Units (SkillUnit | ToolUnit)
  ↓
Agent._build_messages()  → list[Message]
  ↓
AIProvider.generate(messages)  → ProviderResponse
  ↓
ExecutionResult (output + provider metadata)
```

## 3. Breaking Changes

1. **AIProvider.generate() signature changed**: Da `generate(prompt: str) -> str` a `generate(messages: list[Message]) -> ProviderResponse`. Chi implementa `AIProvider` localmente deve adeguarsi.
2. **Agent._build_prompt() rimosso**: Rimpiazzato da `_build_messages()`. Impatto solo su subclassi di `Agent` che ne facevano l'override.

---

## 4. Problemi Rimasti

- Nessun problema bloccante.
- `ProviderManager` non è ancora iniettabile nell'`Agent`; il provider deve essere ancora passato come istanza concreta. Futura estensione.

---

## 5. Compatibilità Futura

- **OpenAI / Gemini / Anthropic**: Basta implementare `AIProvider` con `generate(messages: list[Message]) -> ProviderResponse`. La struttura `Message` è già compatibile con il formato OpenAI Chat Completions.
- **Streaming**: `ProviderConfig` include già il campo `timeout`. Quando si aggiungerà lo streaming, basterà aggiungere un parametro `stream: bool` a `ProviderConfig` e fare l'override di `generate()` con un generatore.
- **Tool Calling nativo LLM**: Il campo `finish_reason` in `ProviderResponse` già restituisce `"tool_calls"` dai modelli OpenAI; l'Agent potrà reagire in base a questo nella v0.10.0.
- **Capabilities**: La base class include `capabilities` property predisposta per future dichiarazioni come `{"tools": True, "vision": True}` per routing intelligente nell'Engine.
