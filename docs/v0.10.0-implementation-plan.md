# Piano di Implementazione — v0.10.0 Planning & Context Runtime

Questo documento definisce il piano ufficiale di implementazione per la milestone **v0.10.0 — Planning & Context Runtime**, con l'obiettivo di evolvere il runtime di Aether da un modello lineare single-pass a un sistema agentico reattivo governato da un loop e da un contesto dinamico.

---

## 1. Obiettivo della Milestone

L'obiettivo primario della milestone v0.10.0 è di introdurre un **Agent Execution Loop** accoppiato a un **AgentContext** mutabile e contratti standardizzati per **ToolCall** e **ToolResult**. Ciò consentirà ad Aether di operare in modo iterativo (ciclo ReAct o simile) permettendo all'LLM di invocare dinamicamente i tool locali, riceverne i risultati, aggiornare la propria memoria e ragionare sul passo successivo, garantendo nel contempo tetti di spesa (token) e protezioni contro i loop infiniti.

---

## 2. Stato Iniziale (v0.9.0)

Attualmente il runtime si basa sui seguenti moduli:
*   `Agent`: Gestisce lifecycle e provider. Prepara l'`ExecutionContext` (immutabile), compila ed esegue il piano, prepara l'array statico di `Message` ed esegue una sola chiamata `generate()` al provider LLM.
*   `ExecutionEngine`: Compila l'`ExecutionPlan` con skill e tool predeterminati staticamente ed esegue le unità in sequenza lineare.
*   `ExecutionPlan`: Lista fissa di unità (`units`). Mantiene lo stato dell'esecuzione e raccoglie i risultati di ciascuna unità.
*   `AIProvider`: Interfaccia messaggi-centrica (`generate(messages: list[Message]) -> ProviderResponse`).
*   `ToolExecutor` & `SkillExecutor`: Gestiscono rispettivamente l'esecuzione dei singoli tool (imperativi) e skill (declarative).
*   `OllamaProvider`: Client reale HTTP senza dipendenze per l'interazione con istanze Ollama locali.

---

## 3. Step Implementativi Ordinati

### Step 1: AgentContext Foundation
*   **Obiettivo**: Creare l'oggetto dinamico e mutabile `AgentContext` che accompagna l'esecuzione del task.
*   **Dettagli**:
    *   Definire la classe `AgentContext` in `src/aether/core/context.py` (o modulo dedicato).
    *   Gestire al suo interno la storia dei messaggi (`messages: list[Message]`), i consumi cumulati di token (`prompt_tokens`, `completion_tokens`, `total_tokens`), lo stato dell'esecuzione corrente, e i tool/skill registrati.
    *   Far sì che l'agente ne detenga l'ownership durante l'intero ciclo di vita di un task.

### Step 2: ToolCall / ToolResult Contracts
*   **Obiettivo**: Creare contratti formali standardizzati per mappare le chiamate ai tool dell'LLM.
*   **Dettagli**:
    *   Definire `ToolCall` in `src/aether/providers/types.py` contenente `call_id: str`, `tool_name: str`, `arguments: dict[str, Any]`.
    *   Definire `ToolResult` in `src/aether/providers/types.py` contenente `call_id: str`, `output: str`, `error: str | None`, `success: bool`.
    *   Modificare `ProviderResponse` affinché contenga un campo opzionale `tool_calls: list[ToolCall] | None`.

### Step 3: Provider Tool Capability Foundation
*   **Obiettivo**: Permettere all'agente di esporre le descrizioni (schemi) dei tool al provider.
*   **Dettagli**:
    *   Aggiungere parametri a `AIProvider.generate()` per accettare una lista di schemi o definizioni di tool (es. formato JSON Schema standard).
    *   Configurare l'agente affinché traduca i tool abilitati in schemi compatibili da passare al provider ad ogni turno di generazione.
    *   Predisporre i metodi di parsing di risposta per estrarre `tool_calls` e popolare correttamente `ProviderResponse`.

### Step 4: Agent Execution Loop Foundation
*   **Obiettivo**: Realizzare il ciclo di esecuzione ReAct guidato dall'agente.
*   **Dettagli**:
    *   Modificare `Agent.execute()` per implementare un ciclo `while` controllato:
        1. Generazione risposta dal Provider LLM.
        2. Rilevamento delle intenzioni: se il provider richiede `tool_calls` (finish reason `tool_calls`), si procede al dispatch.
        3. Esecuzione dei tool richiesti tramite l'engine, inserimento di `ToolCall` e `ToolResult` nella cronologia dell'`AgentContext`.
        4. Ripetizione fino al raggiungimento di una risposta testuale finale (finish reason `stop`) o al superamento del limite di turni.
    *   Introdurre un parametro `max_turns: int = 10` per interrompere l'esecuzione in caso di loop infinito.
    *   Mantenere la retrocompatibilità per task eseguiti in modalità pass-through o privi di tool.

### Step 5: Dynamic Tool Dispatch & ExecutionEngine Integration
*   **Obiettivo**: Aggiornare l'`ExecutionEngine` per processare richieste dinamiche di `ToolCall`.
*   **Dettagli**:
    *   Estendere l'`ExecutionEngine` affinché possa ricevere ed eseguire una lista di `ToolCall` generate a runtime, senza richiedere una pre-compilazione statica dell'`ExecutionPlan`.
    *   Integrare i risultati dei tool e le informazioni di errore (`ToolResult`) affinché vengano mappati su `UnitExecutionResult` coerenti e registrati nel contesto.

### Step 6: Primo Esempio End-to-End
*   **Obiettivo**: Realizzare un test/esempio reale in cui l'LLM decide autonomamente di richiamare un tool.
*   **Dettagli**:
    *   Implementare un tool fittizio o reale (es. `CalculatorTool` o `WeatherTool`) nel registry dell'agente.
    *   Porre all'agente una domanda che richiede obbligatoriamente l'uso del tool (es. "Quanto fa 42 * 123?").
    *   Verificare che l'agente esegua il loop completo, richiami il tool, passi il risultato al modello, e produca la risposta corretta.

## 7. Rischi Architetturali e Mitigazione

*   **Loop infiniti**: Se l'LLM allucina ed emette continuamente chiamate a tool non valide, l'agente potrebbe bloccarsi in un ciclo continuo.
    *   *Mitigazione*: Parametro rigido `max_turns` (default `10`), oltre il quale l'esecuzione viene interrotta forzatamente con un errore.
*   **Gestione dello Stato**: La mutabilità dell'`AgentContext` può introdurre effetti collaterali se non isolata correttamente.
    *   *Mitigazione*: Ogni chiamata ad `Agent.execute()` deve istanziare un contesto fresco e indipendente, senza condividere lo stato conversazionale tra esecuzioni parallele dello stesso agente.
*   **Compatibilità dei Provider**: Non tutti i modelli (specialmente quelli locali più piccoli) supportano nativamente il tool calling o producono JSON valido per `arguments`.
    *   *Mitigazione*: Progettare l'interfaccia affinché possa accogliere una logica di parsing basata sia su tool calling nativo dell'API (es. OpenAI) sia su fallback guidato da prompt di sistema per modelli privi di tale supporto.
*   **Crescita del Contesto e Costi Token**: I cicli iterativi accumulano rapidamente messaggi e risultati dei tool all'interno del contesto conversazionale, aumentando i costi delle API e rischiando di superare la context window.
    *   *Mitigazione*: Implementare una strategia iniziale di logging del consumo cumulato dei token nei metadati e preparare l'architettura a future politiche di troncamento o riassunto (summarization) del contesto.
