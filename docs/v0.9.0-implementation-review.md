## 1. Stato Attuale Provider Architecture

L'attuale astrazione in Aether (introdotta nella milestone `v0.2.0-provider-foundation` e situata in `src/aether/providers/base.py`) definisce un'interfaccia molto basilare:
```python
class AIProvider(ABC):
    @abstractmethod
    def generate(self, prompt: str) -> str:
        pass
```
L'`Agent` (in `src/aether/agents/agent.py`) compila attualmente il context concatenando istruzioni, memoria e output dei tool in un'unica stringa, per poi chiamare `self.provider.generate(prompt)`. 
Esiste una sola implementazione concreta, `MockProvider`, ad uso esclusivo dei test.

## 2. Problemi rispetto a un Uso Reale con LLM

La firma `generate(prompt: str) -> str` Ă¨ incompatibile con le API dei modelli LLM moderni (OpenAI, Anthropic, Gemini, Ollama):
1. **Assenza di Chat History / Messaggi Strutturati**: I LLM lavorano con un array di messaggi dotati di ruolo (`system`, `user`, `assistant`), essenziali per separare le istruzioni di sistema dalle interazioni utente.
2. **Nessun Modello Dati in Input**: Non esiste alcun costrutto per la configurazione del provider (es. `api_key`, `base_url`, `model`, `temperature`, `max_tokens`, `timeout`). L'`Agent` o l'istanza stessa non hanno uno standard di iniezione.
3. **Nessun Modello Dati in Output**: Restituire una stringa non permette di estrarre metadati vitali dalla risposta del LLM (es. `usage` per i token consumati, `finish_reason` per sapere se il modello si Ă¨ arrestato per limiti o per chiamare un tool).
4. **Assenza di Factory / Manager**: Manca un `ProviderManager` che permetta l'inizializzazione dichiarativa (es. da YAML) di un provider in base al nome.
5. **Errore e Resilienza**: I provider sollevano errori disparati (es. HTTP timeouts, Rate Limits). Senza classi di eccezione standardizzate (`ProviderError`, `RateLimitError`), l'Agent non puĂ˛ implementarvi fallback o logica di retry elegante.

## 3. Impatto sul Runtime Esistente

Modificare i Provider comporterĂ  impatti moderati su alcuni core components:
*   **Agent**: Il metodo `_build_prompt` dovrĂ  essere rifattorizzato in `_build_messages` per costruire la lista di messaggi al posto della stringa.
*   **Provider (base.py)**: Cambiamento profondo della firma del contratto.
*   **ExecutionEngine**: Nessun impatto. L'engine rimarrĂ  disaccoppiato in quanto gestisce l'orchestrazione delle Unit prima del provider.
*   **ExecutionResult**: Nessun impatto diretto sull'architettura, ma l'Agent potrĂ  iniettare i nuovi metadata (`usage`, `finish_reason`) provenienti dal provider dentro la mappa di `metadata` del risultato.

## 4. Piano Dettagliato di Implementazione

La nuova architettura si baserĂ  sui seguenti componenti:

### Tipi e Configurazioni (`src/aether/providers/types.py`)
```python
from dataclasses import dataclass, field

@dataclass
class Message:
    role: str
    content: str

@dataclass
class ProviderConfig:
    api_key: str | None = None
    base_url: str | None = None
    model: str | None = None
    temperature: float = 0.7
    max_tokens: int | None = None
    timeout: float = 30.0
    max_retries: int = 3

@dataclass
class ProviderResponse:
    content: str
    model: str
    usage: dict[str, int] = field(default_factory=dict)
    finish_reason: str = "stop"
```

### Contratto Aggiornato (`src/aether/providers/base.py`)
```python
class AIProvider(ABC):
    def __init__(self, config: ProviderConfig | None = None):
        self.config = config or ProviderConfig()

    @abstractmethod
    def generate(self, messages: list[Message]) -> ProviderResponse:
        pass
```

### Gestore Centralizzato (`src/aether/providers/manager.py`)

## 6. Eventuali Breaking Change

1.  **AIProvider Contract**: Chi implementava localmente `AIProvider` dovrĂ  passare da `generate(prompt: str) -> str` a `generate(messages: list[Message]) -> ProviderResponse`.
2.  **Agent Internals**: Il metodo protetto `Agent._build_prompt` viene rimosso in favore di `Agent._build_messages`. (Non rompe l'API pubblica, ma ha impatto sulle subclassi dell'Agent che ne effettuavano l'override).
