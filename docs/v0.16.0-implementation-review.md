# Aether v0.16.0 - Implementation Review
## Agent Intelligence Layer

### Architettura Implementata
La milestone v0.16.0 introduce il primo ciclo cognitivo esplicito per Aether, posizionando il layer di `Planning` al di sopra del pre-esistente `ExecutionEngine`, come definito nell'Architecture Review.
Il flusso implementato nel metodo `Agent.achieve(Goal)` è il seguente:
`Goal -> Planner -> CognitivePlan -> Conversione -> engine.ExecutionPlan -> ExecutionEngine -> Observation -> Planner -> Decision`

L'`Agent` delega interamente la logica di ragionamento al `Planner` instanziato, mentre l'`ExecutionEngine` esegue i passi risultanti ignorando la sovrastruttura cognitiva.

### Nuovi Modelli Introdotti
Il modulo `src/aether/planning/types.py` implementa rigorosamente le direttive con l'utilizzo di `@dataclass(frozen=True)`:
- **`Goal`**: Astrazione immutabile dell'obiettivo contenente `description`, `success_criteria`, `constraints`.
- **`CognitivePlan`**: Rappresentazione strategica generata dal Planner contenente `plan_id`, un riferimento al `Goal`, e la tupla `steps` (azioni logiche, non le unit meccaniche del motore).
- **`Observation`**: Modello immutabile per incapsulare i feedback retroattivi dall'`ExecutionEngine` associandoli univocamente a un `plan_id` e a uno `step_id`.
- **`Decision` & `DecisionAction`**: Verdetto del planner post-osservazione. Può emettere `CONTINUE`, `REPLAN`, oppure `FINISH`.

### Decisioni Prese
1. **Risoluzione Nomenclatura Piani**: Al fine di non inquinare l'esistente `aether.engine.plan.ExecutionPlan`, il modello cognitivo è stato nominato `CognitivePlan`. L'`Agent` converte trasparentemente un `CognitivePlan` in un `ExecutionPlan` prima del dispatch all'`ExecutionEngine`, mantenendo una netta separazione tra "intenzione" ed "esecuzione meccanica".
2. **Minimal Planner**: È stata introdotta l'interfaccia `BasePlanner(ABC)` e un'implementazione funzionale `BasicPlanner` senza alterare in modo distruttivo il vecchio loop ReAct dell'Agente, che rimane confinato nel pre-esistente `execute()`. `achieve()` accoglie la nuova via maestra.

### Test Eseguiti
Sono stati implementati unit tests dedicati all'interno di `tests/test_planning.py`.
- **Validazioni Immutabilità**: Testata `FrozenInstanceError` sui modelli `Goal` e `CognitivePlan`.
- **Planner Logic**: Validazione del `BasicPlanner` sia in assenza di provider (fallback deterministico) sia in presenza di provider mock, valutandone le decisioni (`FINISH` su successo, `REPLAN` su errore in Observation).
- **Lifecycle Integration**: Test dell'effettivo completamento del ciclo vitale e popolamento di `metadata` chiamando `Agent.achieve()`.
- **Regression**: Tutto l'eseguibile ha superato il comando `pytest tests/ -q -W error` garantendo 0 breaking changes (151 tests passed, 1 skipped).

### Limitazioni e Lavori Futuri
- Il `BasicPlanner` attuale è inteso come architettura fondazionale. La conversione da "step cognitivo logico" a `SkillUnit`/`ToolUnit` all'interno dell'`Agent.achieve()` è al momento molto embrionale (mappa genericamente sulle skill disponibili) ed è concepita per essere scalata nelle milestone future, contestualmente all'integrazione di strategie Planner avanzate (come ReAct o OODA).
- L'utilizzo nativo di Structured Outputs (JSON schemi) è stato escluso per evitare over-engineering immediato dei Provider esistenti; seguirà nella v0.17.
