# Architecture Review — v0.14.0 Parallel Execution Runtime

**Data:** 2026-07-18  
**Stato:** Proposta Architetturale  
**Milestone di riferimento:** v0.14.0 (Parallel Execution Runtime)  
**Autore:** Senior Software Architect

---

## 1. Obiettivi

La milestone **v0.14.0** estende le capacità multi-agent di Aether introducendo l'esecuzione parallela e concorrente di deleghe a più sotto-agenti, superando il limite dell'esecuzione puramente sequenziale introdotta nella v0.13.0.

Obiettivi specifici:
1.  **Parallel Execution Layer**: Consentire l'invocazione simultanea di più agenti.
2.  **ThreadPool-based Orchestration**: Eseguire agenti multipli in parallelo sfruttando un pool di thread locali, senza alterare la natura sincrona della codebase.
3.  **Timeout & Retry Management**: Introdurre policy configurabili per la gestione di scadenze temporali e tentativi automatici di esecuzione.
4.  **Event & State Synchronization**: Integrare l'emissione di eventi ed il tracciamento dei task paralleli in modo sincrono e isolato.

---

## 2. Stato Attuale (Post v0.13.0)

Il runtime di coordinamento introdotto nella v0.13.0 è sincrono e sequenziale:
*   Un agente chiama `Coordinator.delegate()`, che sospende l'esecuzione dell'agente chiamante finché l'agente figlio non ha completato `execute()`.
*   Non è possibile inviare compiti in parallelo a più agenti subordinati (es. richiedere analisi contemporanee a due analisti diversi).

---

## 3. Scelta Tecnologica: asyncio vs ThreadPoolExecutor

Per implementare l'esecuzione parallela nel runtime di Aether si sono valutate due strade:

### Approccio A: asyncio Nativo
*   *Vantaggi*: Modello di concorrenza standard a livello di linguaggio per operazioni I/O-bound.
*   *Svantaggi*: Rendere Aether asincrono comporterebbe la riscrittura di tutte le firme e i loop chiave (da `Agent.execute` a `ExecutionEngine.run`, fino ai provider e ai tool) con la keyword `async/await`. Questo distruggerebbe la semplicità della codebase e la retrocompatibilità totale con le milestone v0.10–v0.13, violando il vincolo di stabilità dell'Agente.

### Approccio B: ThreadPoolExecutor Locale (Scelta Approcciata)
*   *Vantaggi*: Permette di mantenere le API sincrone esistenti intatte. Poiché le chiamate degli agenti sono prevalentemente I/O-bound (chiamate HTTP verso provider LLM quali Ollama/OpenAI, accessi a file o DB locali), il GIL di Python non costituisce un limite reale alla concorrenza. I thread condividono nativamente lo spazio di memoria, consentendo il riutilizzo di `AgentRegistry`, `TaskTracker`, ed `EventEmitter` in-process.
*   *Svantaggi*: Richiede cautela sull'accesso a risorse condivise non thread-safe (es. connessione SQLite).
*   *Motivazione*: L'approccio con **`ThreadPoolExecutor` locale** rispetta appieno la filosofia Aether di minimalismo, zero over-engineering e retrocompatibilità al 100%.

## 4. Nuovi Componenti e Responsabilità

L'architettura prevede l'introduzione di nuovi componenti inseriti in `src/aether/coordination/` o l'estensione dei contratti esistenti:

### A. ParallelCoordinator (Estensione di `Coordinator`)
Sviluppato come subclass o estensione naturale di `Coordinator`.
*   **Responsabilità**:
    *   Esporre il metodo `delegate_parallel(self, delegations: list[dict[str, Any]], parent_task_id: str | None = None, timeout: float | None = None) -> list[ExecutionResult]`.
    *   Ciascuna delega nella lista descrive l'agente target (`agent_name`), le istruzioni (`instruction`), ed eventuali parametri di override (es. retry policy).
    *   Gestire il ciclo di vita del pool tramite `ThreadPoolExecutor`.

### B. Parallel Execution Helpers (`src/aether/coordination/parallel.py`)
*   **RetryPolicy**: Contratto dati per configurare i tentativi (es. `max_retries`, `backoff_factor`).
*   **ExecutionTaskWrapper**: Funzione interna delegata al singolo thread che incapsula la logica di retry ed esecuzione per evitare corruzione dello stato del thread chiamante.

---

## 5. Context Isolation, Eventi e Tracciamento dei Task

### Context Isolation
Ogni thread esegue in isolamento:
*   Viene generato un `task_id` univoco per ciascun sotto-task.
*   La memoria a breve termine (`ConversationMemory`) isola i messaggi indicizzandoli per `task_id`, garantendo che i thread non si contaminino a vicenda.

### Integrazione con TaskTracker ed EventEmitter
L'integrazione avviene in modo naturale e senza lock complessi:
*   **TaskTracker**: Ciascun thread chiama `tracker.create` e `tracker.transition` per il proprio `task_id`. Le transizioni di stato avvengono indipendentemente per ciascun task.
*   **EventEmitter**: Gli eventi sono emessi sincronicamente dai singoli thread e gestiti dai listener registrati sul thread principale (sotto la garanzia di sicurezza dei callback di Python).

---

## 6. Gestione Errori Parziali e Gestione Risorse

*   **Errori parziali**: Se alcune deleghe in parallelo falliscono ed altre hanno successo, `delegate_parallel` restituisce l'elenco completo dei risultati (`list[ExecutionResult]`). L'orchestratore chiamante analizzerà i singoli risultati per decidere se procedere o sollevare un errore generale.
*   **Thread Safety su SQLite**: SQLite con `check_same_thread=False` consente l'uso condiviso, ma le scritture contemporanee su `SemanticMemory` potrebbero bloccarsi. La documentazione di implementazione prevederà l'uso di un lock semplice di thread in `SemanticMemory` in caso di accessi concorrenti da esecuzioni parallele.
