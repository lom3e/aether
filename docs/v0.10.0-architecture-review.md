# Architecture Review — v0.10.0 Planning & Context Runtime

**Data:** 2026-07-17  
**Stato:** Proposta Architetturale  
**Milestone di riferimento:** v0.10.0 (Planning & Context Runtime)  

---

## 1. Stato Attuale del Runtime (v0.9.0)

Con il completamento della milestone v0.9.0 (Provider Runtime Integration), la pipeline di esecuzione di Aether si articola sui seguenti componenti principali:

*   **Agent**: Gestisce l'identità (`name`, `role`), lo stato del ciclo di vita (`lifecycle`) e la configurazione del provider. È responsabile di preparare l'`ExecutionContext`, delegare la pianificazione e l'esecuzione all'`ExecutionEngine`, convertire il contesto e i risultati parziali in un array strutturato di `Message`, ed eseguire l'invocazione singola del provider LLM per ottenere la risposta finale (`ProviderResponse`).
*   **ExecutionEngine**: Orchestratore sincrono. Riceve il contesto, compila in modo lineare e deterministico l'`ExecutionPlan` (inserendo le skill associate all'agente ed eventuali tool specificati staticamente nei metadati del task), ed esegue le unità in modalità *fail-fast*.
*   **ExecutionPlan**: Struttura dati che modella una lista ordinata di unità (`SkillUnit` o `ToolUnit`). Mantiene lo stato dell'esecuzione e raccoglie i risultati di ciascuna unità.
*   **AIProvider**: Interfaccia astratta per l'interazione con l'LLM, basata su messaggi (`generate(messages: list[Message]) -> ProviderResponse`).
*   **Skills**: Unità di comportamento dichiarative ed esenti da effetti collaterali esterni, validate rispetto allo stato dell'agente.
*   **Tools**: Unità di esecuzione imperative, dotate di effetti collaterali sul sistema o sulla rete (es. lettura/scrittura filesystem, chiamate API), memorizzate in un registro centrale (`ToolRegistry`).

---

## 2. Limiti Attuali dell'Architettura

Sebbene l'architettura attuale sia solida, ben testata e modulare, presenta limiti fondamentali che ne impediscono l'evoluzione verso agenti autonomi complessi:

1.  **Assenza di un Agent Loop (Ciclo Agentico)**:
    Il flusso di esecuzione è strettamente lineare e a singolo passaggio: `Task` → `ExecutionEngine` → `Provider.generate()` → `Result`. L'agente non ha la capacità di ricevere la risposta dall'LLM, decidere di eseguire un'azione correttiva o un tool, raccogliere l'output e ripresentarlo al modello in un ciclo iterativo (es. pattern ReAct).
2.  **Assenza di Tool Calling Dinamico**:
    L'inserimento di un `ToolUnit` nell'`ExecutionPlan` avviene in modo predeterminato prima di invocare il provider (ad esempio estraendo parametri statici da `task.metadata`). L'LLM non ha modo di decidere *dinamicamente* se e quando invocare uno o più tool sulla base dell'andamento della conversazione.
3.  **Assenza di un Context Object Centralizzato e Mutabile**:
    L'`ExecutionContext` attuale è una struttura immutabile costruita all'inizio dell'esecuzione del task. Non esiste un contenitore di stato (`AgentContext`) a livello di run/sessione in grado di tracciare dinamicamente la storia dei messaggi della conversazione corrente, accumulare i consumi di token e registrare i risultati intermedi dei tool eseguiti all'interno di un ciclo.

## 3. Proposta Architetturale v0.10.0: Planning & Context Runtime

Per superare questi limiti, la milestone v0.10.0 introdurrà i seguenti blocchi architetturali:

```text
       Agent (Coordinatore del Loop) ──[Aggiorna]──> AgentContext (Storia, Stato, Token)
         │                     ▲
         │ (Invia Messaggi)    │ (Ritorna Chiamata Tool / Stop)
         ▼                     │
    AIProvider <───────────────┘
         │
         ▼ (Rileva ToolCall)
   ExecutionEngine ──[Esegue]──> Tools / Skills (Restituisce ToolResult)
```

### A. AgentContext
Un nuovo oggetto mutabile che accompagna la singola esecuzione di un task. Sarà responsabile di memorizzare:
*   La cronologia dinamica dei messaggi (`list[Message]`), inclusi i turni di sistema, i messaggi dell'utente, i pensieri/risposte dell'assistente e i risultati dei tool.
*   I metadati di utilizzo dei token cumulati durante tutte le iterazioni del ciclo.
*   Lo stato corrente del piano e l'inventario dei tool disponibili per l'agente.

### B. Planning Foundation
Evoluzione dell'`ExecutionPlan` in una struttura dinamica in grado di accogliere modifiche a runtime.
*   Introduzione di un meccanismo di pianificazione in cui l'LLM, prima di produrre la risposta finale, può elaborare e proporre una sequenza di passi (piano di azione).
*   L'agente può verificare, validare ed eventualmente correggere il piano in corsa basandosi sull'esito dei singoli step.

### C. Contratti ToolCall e ToolResult
Introduzione di interfacce standardizzate per separare la richiesta di esecuzione di un tool dal suo risultato. Questi contratti appartengono interamente al nucleo di runtime (`src/aether/core/execution.py`) e sono completamente indipendenti ed agnostici rispetto ai provider LLM esterni:
*   **`ToolCall`**: Rappresenta l'intenzione dell'LLM di invocare un tool. Contiene `call_id` (identificativo unico), `tool_name` (nome del tool nel registry) e `arguments` (parametri strutturati, tipicamente dizionario JSON).
*   **`ToolResult`**: Rappresenta l'esito dell'esecuzione. Contiene `call_id`, `output` (contenuto generato o errore come stringa o JSON), e flag di stato (es. successo/errore).

I provider avranno esclusivamente la responsabilità di mappare le proprie strutture dati native (es. OpenAI tool calls o Ollama tool structures) da e verso queste definizioni agnostiche di Aether.


### D. Agent Execution Loop
La logica di esecuzione dell'agente si evolve da una chiamata lineare a un ciclo iterativo controllato:
1.  **Inizializzazione**: L'Agente riceve il `Task`, istanzia un `AgentContext` e vi inserisce il System Message e la prima richiesta dell'User.
2.  **Ciclo**:
    *   Invia la sequenza di messaggi dell'`AgentContext` al `AIProvider`.
    *   Il `AIProvider` restituisce una `ProviderResponse`.
    *   Se `response.finish_reason == "tool_calls"` o sono presenti `tool_calls` nel corpo della risposta:
        *   L'agente estrae le `ToolCall` e le passa all'`ExecutionEngine`.
        *   L'`ExecutionEngine` esegue i tool tramite il `ToolExecutor` e restituisce i `ToolResult`.
        *   L'agente inserisce nel contesto sia il messaggio dell'assistente con le `ToolCall` sia i messaggi di sistema contenenti i `ToolResult`.
        *   Il ciclo riparte per la successiva iterazione.
    *   Se `response.finish_reason == "stop"` o viene prodotta la risposta finale dell'agente:
        *   Il loop termina e viene restituito l'`ExecutionResult` finale.
3.  **Controllo di Sicurezza**: Il ciclo implementa un limite massimo di iterazioni (`max_loops` o `max_turns`, es. default = 10) per prevenire loop infiniti o consumi anomali in caso di allucinazione del modello.

## 4. Separazione delle Responsabilità

Per preservare l'approccio modulare e pulito di Aether, le responsabilità dei tre componenti principali sono così delimitate:

| Componente | Responsabilità Principali | Cosa NON Deve Fare |
| :--- | :--- | :--- |
| **Agent** | - Gestione dell'identità e del lifecycle dell'agente.<br>- Ownership dell'`AgentContext` (conservazione della cronologia messaggi).<br>- Gestione e controllo del loop agentico (ReAct).<br>- Rilevamento delle intenzioni dell'LLM (tool calls vs risposta finale). | - Conoscere i dettagli di implementazione o di I/O dei singoli Tool.<br>- Tradurre direttamente i messaggi in formati specifici di API terze. |
| **ExecutionEngine** | - Esecuzione fisica, sicura e controllata delle unità di lavoro (`ToolCall`, `SkillUnit`, `ToolUnit`).<br>- Validazione dei permessi e vincoli di sicurezza dei tool.<br>- Dispatching delle chiamate agli esecutori specialistici (`ToolExecutor`, `SkillExecutor`). | - Gestire lo stato della conversazione globale o della sessione.<br>- Determinare le scelte strategiche dell'agente o interpretare i messaggi dell'LLM. |
| **Provider** | - Traduzione bidirezionale dei dati di Aether da/verso il protocollo nativo dell'LLM (OpenAI schema, Ollama payload).<br>- Esposizione degli schemi dei tool abilitati nel formato corretto dell'API.<br>- Estrazione dei metadati di risposta (chiamate dei tool nativi, uso token, finish reason). | - Gestire il registro dei tool locali.<br>- Prendere decisioni in merito alle politiche di retry o di timeout delle esecuzioni dei tool. |

## 5. Compatibilità Futura

Il design di questo runtime di pianificazione e contesto garantisce una scalabilità lineare verso le seguenti funzionalità avanzate:

1.  **OpenAI Tools**:
    La struttura standard di `ToolCall` (con `call_id` ed argomenti JSON) garantisce una compatibilità diretta (1:1) con le API dei modelli OpenAI e Anthropic, abilitando l'invocazione parallela dei tool nativi.
2.  **Ollama Tool Calling**:
    L'`OllamaProvider` potrà mappare gli schemi dei tool locali nel parametro `tools` previsto dalle recenti versioni delle API di Ollama (`/api/chat`), decodificando in modo pulito le `tool_calls` restituite senza costringere il core di Aether a gestire parser di testo ad-hoc per i modelli locali.
3.  **Memory Evoluta**:
    La presenza di un ciclo iterativo consente all'agente di interrogare la propria memoria interna (es. vettoriale o a breve termine) nel corso delle iterazioni centrali del loop per recuperare informazioni contestuali rilevanti e, al termine del loop, memorizzare nuovi fatti appresi.
4.  **Multi-Agent Orchestration**:
    Poiché un agente Aether implementa il contratto `Agent.execute(Task) -> ExecutionResult`, esso può essere registrato nel `ToolRegistry` di un altro agente come se fosse un normale tool imperativo. Quando l'agente orchestratore decide di avviare un sotto-task, genera una `ToolCall` diretta all'agente collaboratore; quest'ultimo esegue il suo sotto-loop in un `AgentContext` isolato e restituisce l'output finale come `ToolResult`, realizzando nativamente architetture gerarchiche complesse.
