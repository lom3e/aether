## 1. Obiettivo della Milestone

Il focus della milestone v0.9.0 Ă¨ collegare Aether a LLM reali, evolvendo l'astrazione del Provider da un prototipo string-in/string-out a un runtime robusto in grado di gestire le complessitĂ  dei modelli moderni (messages, configurazioni, metadata, fallback, errori). Alla fine della milestone, l'istruzione `agent.run("Hello")` genererĂ  una chiamata API reale e restituirĂ  una risposta del modello.

## 2. Stato Attuale

- L'interfaccia corrente in `src/aether/providers/base.py` Ă¨ limitata a: `generate(prompt: str) -> str`.
- L'`Agent` costruisce la richiesta concatenando stringhe (task + memory + tool output) tramite `_build_prompt()`.
- Mancano completamente: gestione ruoli (System/User/Assistant), gestione timeout/retry, tracciamento token/usage, registry dei provider (ProviderManager), streaming e standardizzazione degli errori.

---

## 3. Analisi Architetturale e Problemi

Il passaggio a veri LLM (OpenAI, Ollama, Anthropic) espone diversi limiti dell'astrazione attuale:
1. **Model Context**: I modelli conversazionali necessitano di array di messaggi strutturati (ruolo + contenuto) per rispettare le "System instructions" e separarle dall'input utente.
2. **Configurazione Isolabile**: L'Agent non dovrebbe conoscere chiavi API o base URL; la configurazione deve essere passata al layer Provider in modo indipendente.
3. **Risultati Ricchi**: Ritornare solo una stringa impedisce di raccogliere metriche fondamentali come l'utilizzo dei token (costi) o il `finish_reason` (per capire se l'output Ă¨ stato troncato o se il modello richiede una tool call).
4. **Instanziazione Dinamica**: Attualmente il provider viene iniettato manualmente. Serve un `ProviderManager` per istanziare i provider da configurazioni o environment.

---

## 4. Nuova Architettura Proposta

L'architettura aggiornerĂ  l'interfaccia `AIProvider` senza rompere la separazione delle responsabilitĂ  (Agent = Reasoning, Engine = Execution, Provider = IO verso LLM).

### 4.1 Modelli Dati (in `src/aether/providers/types.py`)

```python
from dataclasses import dataclass
from typing import Any

@dataclass
class Message:
    role: str  # "system", "user", "assistant"
    content: str

@dataclass
class ProviderConfig:
    api_key: str | None = None
    base_url: str | None = None
    model: str | None = None
    temperature: float = 0.7
    max_tokens: int | None = None
    timeout: float = 30.0
    max_retries: int = 3

@dataclass
class ProviderResponse:
    content: str
    model: str
    usage: dict[str, int]
    finish_reason: str
```

### 4.2 L'Interfaccia Base (in `src/aether/providers/base.py`)

```python
from abc import ABC, abstractmethod

class AIProvider(ABC):
    def __init__(self, config: ProviderConfig | None = None):
        self.config = config or ProviderConfig()

    @abstractmethod
    def generate(self, messages: list[Message]) -> ProviderResponse:
        pass
        
    @property
    def capabilities(self) -> dict[str, bool]:
        # Foundation for capabilities: tools, vision, streaming
        return {"streaming": False, "tools": False}
```

### 4.3 Gestione Errori (in `src/aether/providers/errors.py`)
Creare eccezioni custom per uniformare i fallimenti trasversalmente a tutti i provider:
*   `ProviderError` (Base)
*   `RateLimitError`
*   `TimeoutError`
*   `AuthenticationError`

### 4.4 ProviderManager (in `src/aether/providers/manager.py`)
Un registry pattern per instanziare i provider in base al nome (es. `manager.get("openai", config)`).

### 4.5 Modifiche all'Agent (in `src/aether/agents/agent.py`)
Aggiornare l'Agent per costruire la history dei messaggi anzichĂ© una stringa:
- Convertire `_build_prompt` in `_build_messages() -> list[Message]`.
- Mappare `Agent.role` nel `Message(role="system")`.
- Estrarre il `content` dalla `ProviderResponse` prima di inserirlo nell'`ExecutionResult`.

### 4.6 Implementazioni Real-World
Implementeremo un `OllamaProvider` (per dev/test locale gratuito) e un `OpenAIProvider` come foundation.

## 9. Rischi e Mitigazioni

*   **Rischio**: L'inclusione di SDK di provider (es. `openai` package) potrebbe appesantire le dipendenze core.
    *   **Mitigazione**: Utilizzare client HTTP standard (es. `httpx` o `requests`) per i provider semplici come Ollama. Per SDK complessi, importarli dinamicamente o renderli dipendenze opzionali (es. `pip install aether[openai]`).
*   **Rischio**: Backward compatibility interrotta.
    *   **Mitigazione**: `Agent.execute()` continuerĂ  a ritornare `ExecutionResult`. L'unico vero breaking change Ă¨ interno all'implementazione personalizzata dei Provider, mitigato aggiornando `MockProvider`.

## 10. CompatibilitĂ  Futura
